## 2. Что берём из sitebuilder **как есть (идея, не код)**

Кода `sitebuilder` — CJS, vanilla JS, нет типов. Переносим **паттерны**, а не файлы:

| Паттерн | Источник | Зачем |
|---|---|---|
| **Версионирование внутри секции (v1/v2/v3 при одинаковом контрактe данных)** | `section/*.cjs` | Ключевой источник визуальной вариативности — ложится прямо в §3.4 архитектуры. |
| **CSS-переменные как единственный источник стиля секций** | `common/style.cjs` | Секции не знают цветов, все секции совместимы с любой темой. |
| **`decoreSection` идея связи версий** | `builer.cjs:35–39` | Примитивная, но в seobuilder3 перерастёт в систему compatibility-правил. |

**Не берём из sitebuilder**:
- Контракт данных секций (короткий маркетинговый, неподходящий под SEO).
- Весь контент-пайплайн (заменяется на пайплайн seobuilder2).
- Фиксированный порядок секций на одной странице.
- Mobile-checkbox hack (на TS/ESM есть лучше).
- Структуру деплоя (её нет).

## 3. Что адаптируем (не ложится напрямую)

### 3.1. Контракт данных секций — расширяется под SEO

Контракт sitebuilder (`hero.title/subtitle`, `features.items[{title,text}]`) слишком беден. В seobuilder3 каждая секция должна уметь:

- Длинный html-фрагмент (абзацы, `<h3>`, `<ul>`, таблицы, FAQ) — для категорий, статей, хабов.
- Опциональный «rich» контейнер с внутренней перелинковкой (`<a href="/{lang}/category/…">`).
- Медиа (image, srcset, lazy) + alt.
- Валидация через zod-схему в ран-тайме.

Практическое следствие: **у каждой секции будет два режима — `marketing` (короткий, для hero/cta/testimonials) и `longform` (для категорий/статей)**. Это не две разных секции, а переключаемый fallback внутри одного шаблона. Альтернативы см. в §5.

### 3.2. Типы страниц seobuilder2 получают шаблонное разнообразие

Сейчас в seobuilder2 `buildSite()` знает каждую страницу по имени (`index.html`, `features.html`, …) и жёстко вызывает соответствующие секции. В seobuilder3:

- **Страница = декларативный список ссылок на шаблоны секций** (`homepage-layout-A.json` → `["hero:v2", "value:v1", "features:v3", "faq:v1", "cta:v2"]`).
- **Шаблонов страниц (page-level layouts) — не 5, а десятки**; каждый тип страницы (homepage / category / article / blog-index / service) имеет пул своих layout-манифестов.
- **Выбор конкретного манифеста — детерминированный, от seed = `hash(domain + page_slug)`**.

Это даёт на одну «главную» десятки разных компоновок без размножения кода, и эти компоновки считаются не «random», а «seeded from site identity».

### 3.3. Темы

В seobuilder2 — 8 фиксированных палеток, все dark. В seobuilder3:

- Палетка = **генерируемая**, не захардкоженная: берём базовый hue (seed), производим OKLCH-шкалу (12 оттенков), проверяем контрастность по WCAG.
- Добавляем ортогональную ось **режима** (light / dark / duotone) — даёт × 3 к вариативности без кода.
- **Типографика** = пара «heading / body» из пула, включая serif+sans-комбо (в seobuilder2 только sans).
- **Layout tokens** (`radius`, `spacing`, `container-width`, `grid-gap`) — отдельный seed.

Темы и шаблоны комбинируются **независимо** — шаблон не имеет права хардкодить цвета/размеры.

### 3.4. Внутренняя перелинковка

В seobuilder2 словарь `relatedPages` — статичный, на 5–7 страниц. В seobuilder3 появляются **категории + десятки статей**, а значит:

- Перелинковка строится **постфактум** на готовом наборе страниц (граф).
- Правила: `article → category` (каноническая связь), `article → сиблинг-статья той же категории` (N=3), `category → pillar-статья` (curated), `homepage → top-N категорий`.
- Детерминированно от seed, с ограничением на общее число исходящих ссылок с одной страницы (SEO-гигиена).

### 3.5. Блог

seobuilder2 генерит 3–5 статей из пула 40 топиков. Для seobuilder3:

- Пул топиков **параметризуется по тематике сайта** (не 40 универсальных, а генерится под keyword сайта).
- Длина — 800–2500 слов (не фиксированная).
- Шаблонное разнообразие — каждая статья получает свой **article-layout** (hero + intro + TOC + sections[] + FAQ + related), и каждая секция внутри — свой v1/v2/v3.

### 3.6. Категории / хабы

Новый тип страницы, которого нет ни в одном исходнике. Нужен:

- Hero категории (с ключевыми словами категории).
- Обзорный текст (longform).
- Список статей категории (grid/list/featured variants).
- Перелинковка на соседние категории (breadcrumbs + siblings).

## 4. Что пишем с нуля

| Компонент | Почему с нуля |
|---|---|
| **Типизированный контракт контента (zod)** | В исходниках нет. SEO-качество без валидации построить нельзя. |
| **Реестр шаблонов (template registry) с декларацией контракта и совместимости** | Ни в одном проекте нет. Без него нельзя безопасно композировать. |
| **Seed-движок (site-level, page-level, section-level)** | В исходниках нет. Нужен для воспроизводимости и для уникальности двух соседних сайтов. |
| **Композитор страницы (picker)** — по pool of layouts + compatibility rules + seed | Новая сущность. |
| **Граф перелинковки** | Новая сущность. |
| **Генератор палитры от hue (OKLCH + WCAG check)** | В seobuilder2 только 8 захардкоженных, не масштабируется. |
| **Артефакт билда + манифест сайта** | Нужен для дебага, аудита и CDN-деплоя. В исходниках — только `_preview/`. |
| **CDN-ориентированный деплой вместо per-domain nginx vhost** | Для масштаба от ~100 сайтов. |
| **Валидатор SEO (самопроверка на рендере)** | В `seo-check.cjs` есть идея — формализовать. |
| **Telegram-бот на TS/ESM с тем же UX команд** | Переписать, не транслитерировать CJS. |

## 5. Конфликты архитектур и развилки

Пункты, по которым оба проекта решают одно и то же по-разному. Даю рекомендацию; финальное слово за владельцем.

### 5.1. Шаблонизация

- **seobuilder2**: строковая конкатенация в функциях. Плюс: быстрее и проще. Минус: нет type-safety, легко XSS.
- **sitebuilder**: то же самое.

**Рекомендация**: выбрать **один** из трёх — (a) `preact` SSR в HTML-строку (типобезопасно, знакомо, JSX), (b) `lit-html`/подобный tagged-template с escape по умолчанию, (c) собственный `html` tagged template c авто-escape. Рекомендую (a) `preact` — даёт полный JSX, компоненты, `renderToString`, тесты тривиальны. 40 KB в dev, 0 в production артефакте (мы рендерим сразу в HTML).

### 5.2. Формат шаблона секции

- **sitebuilder-style**: одна функция `(content, version, theme) => html` с switch по version.
- **seobuilder3-альтернатива**: каждая версия — отдельный файл/модуль; реестр собирает их в `Map`.

**Рекомендация**: второй вариант. Даёт lazy-import, код-сплиттинг при необходимости, проще расширять, естественные тесты per-версии.

### 5.3. Стили

- **seobuilder2**: единый `styles.css`, инъекция темы через CSS-переменные в `:root`.
- **sitebuilder**: то же самое, темы — переменные.

Все сходятся. **Сохраняем**. Ни Tailwind, ни CSS-in-JS в артефакте не нужны (лишний вес и маркер генератора).

### 5.4. Контент-хранилище

- **seobuilder2**: `content-{lang}.json` предвычислено.
- **sitebuilder**: JS-объект `Content.cjs`, перезаписывается каждый раз.

**Рекомендация**: seobuilder2-путь, но **JSON → zod-валидированный TS-тип**. Хранить per-site в `sites/{domain}/content/{lang}.json`.

### 5.5. Деплой

- **seobuilder2**: per-domain nginx vhost + SFTP каждого сайта.
- **sitebuilder**: локальный `public/` без деплоя.

**Рекомендация**: **CDN-friendly артефакт + wildcard host**. На первом этапе сохраняем SFTP-путь seobuilder2 для быстрого запуска; вторым этапом — перевод на object storage (S3-совместимое) + CDN с Host-based роутингом. Это требование «10k сайтов» из ТЗ.

### 5.6. Queue vs sequential для AI

- seobuilder2 делает всё sequential внутри одного run, параллелит только параллельные шаги пайплайна.
- На 100 сайтов в день это норма. На 1000 — нужен внешний очередник (Redis/SQS) + воркеры.

**Рекомендация**: начать с sequential (простота), но заложить интерфейс `Job` и `JobRunner` с прицелом на воркер-модель. Не городить распределёнку до появления реальной нагрузки.

### 5.7. Блог: общий vs per-site

- В seobuilder2 каждый сайт получает свой блог с нуля.
- Альтернатива — shared pool статей с per-site rewriting.

**Рекомендация**: **per-site с уникальным топик-пулом под тематику сайта**. Sharing — путь к дубль-контенту и риску фильтра.

### 5.8. Где развилки требуют решения владельца

| Развилка | Варианты | Рекомендация |
|---|---|---|
| **Рендерер: preact vs tagged-template vs свой** | см. §5.1 | preact SSR |
| **Хостинг артефактов: SFTP+nginx (быстрее стартовать) vs S3+CDN (масштаб)** | оба валидны | Начать SFTP, на этапе 3 (см. roadmap) переключить на CDN |
| **Формат конфигурации сайта: один `site.yaml` vs разнесённые JSON** | оба валидны | Один `site.yaml` на входе, но `content-{lang}.json` остаётся JSON (AI-дружественный формат) |
| **Telegram-бот: отдельный процесс или внутренний worker** | отдельный (как сейчас) vs integrated | Отдельный, как в seobuilder2 — проще перезапускать независимо |
| **Стоит ли в seobuilder3 сохранять backlinks + YouTube интеграции из seobuilder2 сразу** | да / позже / никогда | Позже (этап 4). Сначала ядро. |

По каждой развилке жду решения перед реализацией. До этого момента в архитектуре (03-architecture.md) я фиксирую **рекомендуемый** вариант и помечаю его как «default until decision».
